03 — Concierge Logic

Signals, timing,
and the rules that govern both

The concierge earns trust through precision. These are the explicit rules governing when it speaks, what data it monitors, and how it hands off between surfaces without losing context.

📅 Notification Timing
  • T–5 daysOne morning brief: baggage policy, entry requirements, airline app prompt.
  • T–24 hrsCheck-in alert fires exactly when airline API opens window. Not before.
  • T–4 hrsMorning brief: weather, boarding pass status, hotel info, car rental reminder.
  • T–3 hrsRideshare prompt with live traffic + estimated Uber/Lyft cost.
  • T–1.5 hrsTSA wait alert if delay risk. Gate info if assigned by airline.
  • LandingBaggage carousel push within 90 sec of wheels-down via FlightAware.
  • Never3am pushes. Duplicate information. Unprompted follow-up messages.
📡 Data Signals & Sources
  • FlightFlightAware / OAG: real-time delay, gate, boarding, wheels-down.
  • Check-inAirline APIs. Apple Wallet PassKit event = confirmed. No push needed.
  • LocationGeofence triggers: home airport arrival, terminal exit, destination arrival.
  • HotelPMS integration or polling for room-ready and digital key issuance.
  • TrafficMaps ETA API polled at T–4hrs, dynamically updated on conditions.
  • RideshareUber/Lyft API: live ETA + price pre-loaded with airport address.
  • RentalCar confirmation API: pickup lot, stall, shuttle interval.
🔁 Seamless Segue Rules
  • StateAfter any deep-link, concierge thread resumes on return — no re-context needed.
  • Context"Got Bags" tap dismisses baggage step and loads car rental shuttle card.
  • ChatChat auto-loads current trip phase. Traveler never re-states their situation.
  • Live Act.Live Activity state machine runs on same signals, independent of app state.
  • OTAOTA module refreshes on phase state. Links open post-booking screens directly.
🔒 Privacy & Fallback
  • LocationGeofencing is opt-in. Fallback: time-based triggers using departure time minus travel estimate.
  • KnownConcierge never asks what it already knows. Wallet pass added = confirmed.
  • Quiet hrsPush suppressed 10pm–7am local time. In-app content always available.
  • ReminderDismissed push with still-relevant trigger fires once more at T–2hrs. Never again.
  • FailIf data unavailable: "Couldn't confirm gate — want me to check the airline site?"

05 — Disruption Flows

When things go wrong,
the concierge becomes an agent

Disruption handling is the highest-leverage feature for loyalty and repeat use. A traveler who is rescued from a cancelled flight by a concierge that presents solutions, handles rebooking, and keeps their travel companions informed will return. The system must distinguish between what it can resolve programmatically, what it can assist with via a drafted message, and when it must escalate to a human — and be honest about which situation it's in.

Disruption Decision Framework
Disruption Event Detected
Flight cancelled — DL 284 SEA → SFO
Source: Airline API push at 6:47am. Reason: Mechanical. Aircraft grounded. No replacement scheduled. Concierge triggered across all active surfaces within 60 seconds of confirmed cancellation signal.
Can it be resolved programmatically?
Path A — Programmatic Resolution
Concierge can rebook directly
  • Airline API supports automated rebooking for ticket class
  • No fare difference or difference is pre-authorized
  • Seat availability confirmed on alternative flights
  • Concierge presents ranked options with details
  • User selects, concierge executes, sends updated boarding pass
  • Multi-traveler group notified with acknowledgment request
  • Live Activity updates to new flight data automatically
Path B — Non-Programmatic Escalation
Human resolution required
  • Airline requires payment verification for fare difference
  • Ticket class has rebooking restrictions
  • Travel insurance claim may be triggered
  • Concierge clearly explains why it cannot resolve
  • Concierge drafts a message to airline customer service
  • User reviews and edits the draft before sending
  • Concierge provides direct phone numbers and estimated wait times
  • If at airport: directs to nearest staffed rebooking desk
Agent-Style Option Presentation — Path A

The concierge presents ranked, concrete options — not just a notification. Options are ordered by soonest departure and seat availability. User selects one action; concierge executes. User must acknowledge resolution to close the loop and provide implicit feedback to the concierge learning system.

After the traveler selects an option or acknowledges the disruption, the concierge records the response. 06 — Multi-Traveler System

Group travel with
clear roles and real-time awareness

When a booking includes multiple travelers, the concierge needs to serve everyone while maintaining clear authority about who can act. Notifications flow to all. Changes are made only by the booking owner. Acknowledgment is visible across the group — so nobody gets left behind at the gate because they missed a notification.

Permission Model — V1
Booking Owner
Full agent capability
  • Receives all proactive notifications and disruption alerts
  • Can accept or reject disruption resolutions
  • Can initiate rebooking, room changes, or rental modifications
  • Sees acknowledgment status of all travelers in group
  • Can manually mark a non-app traveler as acknowledged
  • Can send reminder to unacknowledged travelers (one tap)
  • Can add optional note when manually acknowledging ("Texted her — confirmed")
  • Cannot see other travelers' personal account data
Fellow Traveler (App User)
Informed, acknowledgment-capable
  • Receives all relevant notifications for shared itinerary
  • Can acknowledge notifications via push action button (no app open required)
  • Can view their own booking details in-app
  • Can chat with concierge for their own questions
  • Cannot make changes to the shared booking
  • Cannot see other travelers' acknowledgment status
  • Cannot initiate disruption resolutions
  • Does not see the booking owner's full concierge view

07 — Personalization & Learning

A concierge that
gets smarter every trip

The concierge learns from every trip — passively through observed behavior, and explicitly through surfaced moments where it shares what it's noticed. Both matter. Passive learning builds accuracy. Explicit learning builds trust. The traveler always has a clear view of what the concierge has learned and can edit or disable any preference.

📡 Passive Learning Signals
🪑
Seat Preference
Learned from boarding pass patterns across trips. Window vs. aisle, front vs. back of cabin, specific rows avoided.
🧳
Baggage Behavior
Traveler consistently checks bags vs. carry-on only. Pre-pays for bags vs. at counter. Learned after 2 consistent trips.
🚗
Rideshare vs. Drive
Learns from response to rideshare prompts. If always dismissed, rideshare prompt cadence reduces. If always tapped, prompt is surfaced earlier.
🏨
Hotel Preferences
Loyalty program tier, consistently requested floor/view, early check-in success rate, late checkout frequency.
⏰
Timing Preferences
How early traveler typically arrives at airport. Response time to push notifications. Preferred notification timing windows.
🔄
Disruption Choices
Which rebooking option was selected when cancelled. Refund vs. rebook preference. Airline flexibility for fare differences.
💬 Explicit Learning Moments
✨
Surfaced Sparingly
Explicit learning moments are shown at most once per trip, never mid-disruption. Timing matters — post-flight or post-hotel check-in are ideal moments.
🗣
Conversational Framing
Never "You have new preferences to configure." Instead: "I noticed you always prefer window seats — should I prioritize those going forward?"
⚡
Single-Tap Confirmation
Yes / Not Always / No — three answers. No forms. No settings pages. The memory view is where users go to manage; the explicit moment just creates the memory.
🔒
Transparent Memory
Every learned preference is visible in the Memory view. Users can edit values, disable specific signals, or reset the learning model entirely.
Example · Post-flight explicit moment
I noticed you've checked bags on your last 3 trips. Should I skip the carry-on packing tips going forward and go straight to baggage pre-payment options?
Yes, remember this
Not always
User-Facing Memory View

The memory view is the concierge's "what I know about you" panel. It's the transparency layer that makes personalization feel trustworthy rather than surveillance-like. Every preference has a source, a toggle, and an edit option.

08 — International Expansion

V1 is domestic.
V2 is built on its foundation.

International travel isn't a parallel product — it's an additive layer on a well-designed V1 data model. The architecture decision that makes this possible: V1 should carry schema fields for passport, visa, entry requirements, and customs from day one — even if those fields are unpopulated for domestic trips. When international travel is enabled, it's a data fill, not a rebuild.

Roadmap — V1 Domestic → V2 International
V1 — Domestic
Core concierge established
  • US domestic flights only
  • TSA PreCheck / CLEAR prompts
  • Domestic baggage rules by airline
  • USD currency only
  • No customs or immigration flows
  • Schema carries international fields — empty
  • Passport field present, not required
  • Entry requirements field: null
  • Customs form field: null
  • Currency field: defaults to USD
V2 — International Core
International flights & requirements
  • Passport expiry check at booking T–90 days push
  • Visa requirement lookup by destination + nationality
  • ESTA / ETA prompt where applicable
  • Customs and immigration flow at arrival
  • Destination currency and exchange rate
  • International roaming / SIM card prompt
  • Health requirements (vaccines, entry docs)
  • Travel advisory status at booking and T–5 days
  • International baggage rules by route
  • Connecting international flight handling
V3 — International Full
Deep international concierge
  • Multi-leg international connection management
  • Digital customs form pre-fill (ESTA, CBP One)
  • Airport transit visa requirements for connections
  • Trusted Traveler programs: Global Entry, NEXUS
  • International lounge access by card / status
  • Language-aware destination briefing
  • Multi-currency hotel / car billing clarity
  • International disruption: re-entry complications
  • Localized emergency numbers and consulate contacts
  • International travel insurance integration

Passport expiry check (6-month validity rule). Visa requirement by destination and nationality. ESTA for visa-waiver countries. ETA for Canada, UK, etc. Application links and timing.

Destination currency and live exchange rate. Credit card foreign transaction fee warning. ATM vs. exchange bureau guidance. Dynamic currency conversion warning at point of use.

09 — Design Principles

Eight rules that
govern every decision

01
Don't ask what you already know
If boarding pass is in Wallet, don't ask about check-in. If location places the traveler past the carousel, don't prompt about bags. The concierge monitors signals and only asks when genuinely uncertain.
02
One voice across all surfaces
Push, chat, Live Activity, and OTA module are driven by the same state machine. The traveler should never receive contradictory or duplicate information. One voice, everywhere, correctly.
03
Context never resets
After opening Uber, the concierge picks up where it left off. After deep-linking anywhere, returning to the app resumes the flow. The traveler never has to re-explain where they are or what they're doing.
04
Timing is the product
The right information at the wrong time is noise. Every trigger time is an explicit design decision, not a default. The rideshare prompt 30 minutes too early causes panic. 20 minutes too late is useless.
05
In disruption, act — don't just inform
A notification that says "your flight is cancelled" without options is worse than silence. The concierge presents ranked, actionable choices. It always has a next step ready, even when that step is helping a human agent help you.
06
Never send without permission
The concierge can draft messages, suggest actions, and prepare communications — but never sends anything on the traveler's behalf without explicit review and approval. This is especially critical for escalation messages to airlines and hotels.
07
Personalization must be transparent
Every preference the concierge learns is visible and editable. Personalization that feels invisible feels like surveillance. Personalization that explains itself feels like a service. The memory view is not optional — it's foundational to trust.
08
Build for international from day one
The data model carries international fields in V1 even if they're unpopulated. International is not a new product — it's V1 with the fields filled. Every architectural decision should be made with this in mind.
